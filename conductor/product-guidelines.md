# Product Guidelines: SpringFlow

## 1. Documentation & Communication Style
Our communication should balance technical depth with ease of use:
*   **Technical & Precise:** Provide accurate descriptions of architectural patterns, dynamic class generation (Byte Buddy), and mapping logic (MapStruct).
*   **Approachable & Example-Rich:** Every feature must be accompanied by a clear Java and/or Kotlin code snippet. We explain "why" as much as "how."
*   **Concise & Action-Oriented:** Documentation should prioritize getting the developer to a working solution quickly. Use "Quick Start" blocks and TL;DR summaries for complex topics.

## 2. Brand Messaging Pillars
*   **Simplicity & Efficiency:** "One annotation, complete API." We sell the reduction of boilerplate and the acceleration of development cycles.
*   **Flexibility & Control:** Magic should never be a black box. We emphasize that SpringFlow is designed to get out of the way as soon as a custom requirement emerges.
*   **Production Readiness:** SpringFlow isn't a toy. We focus on its adherence to industry standards (OpenAPI 3, JSR-380, Spring Data) and its robust handling of edge cases like security and validation.

## 3. Visual Identity & Design Principles
*   **Clean & Modern:** Documentation (MkDocs) and future UI components should use minimalist aesthetics with a focus on high-contrast code blocks and clear hierarchy.
*   **Spring-Inspired Palette:** Utilize a color scheme compatible with the Spring ecosystem (Greens, Whites, and Dark Greys) to provide a sense of familiarity and integration.
*   **Functional Density:** Prioritize information density over whitespace in technical sections. Developers should be able to find API parameters and configuration keys with minimal scrolling.
